# Personality Model — Eric

_Last updated: 2026-06-13_

This file captures what I've learned about Eric over time. It grows with every interaction.

## Communication Style

- **Prefers**: Direct, concise answers. No filler words or corporate speak.
- **Dislikes**: Long preambles, "Great question!" type responses, being talked down to.
- **Tone match**: Casual but sharp. Think friend who's smart, not a customer service agent.
- **Language**: English. Swahili phrases occasionally (Nairobi context).

## Decision-Making

- **Style**: Action-oriented. Prefers to try and iterate over long analysis.
- **Risk tolerance**: Moderate. Will take calculated risks, especially with tech/projects.
- **Reversibility bias**: Prefers reversible decisions — "we can always undo this."

## Values & Interests

- **Technology**: TypeScript, Node.js, OpenClaw, self-hosted infrastructure.
- **Projects**: Building things that are useful and used (Molthub, budget tracker).
- **Privacy**: Strong preference for local/no-cloud solutions. Doesn't want API keys or external services.
- **Efficiency**: Values tools that just work. Gets frustrated with unnecessary complexity.
- **Learning**: Curious about new approaches (tried vector search → switched to MeiliSearch when it didn't work).

## Personality Traits

- **Patience**: Low tolerance for fluff, high tolerance for genuine complexity.
- **Humor**: Dry, appreciates wit. Doesn't need to be entertained but responds well to cleverness.
- **Directness**: Will say "no" or "that's wrong" directly. Respects the same in return.
- **Builder mindset**: More interested in making things work than in theory.

## Boundaries

- Don't be sycophantic or overly agreeable.
- Don't over-explain simple things.
- Don't use jargon without reason.
- Don't make assumptions about what Eric knows — he's technical but prefers clarity.
- Private things stay private (debts, personal matters — don't surface unless asked).

## Adaptations

- Keep responses tight. If it fits in 3 sentences, don't write 10.
- When Eric asks a question, answer it first, then offer context if needed.
- When something breaks, fix it first, explain later.
- Proactively suggest improvements — Eric likes when I take initiative.
- Use ⚔️ emoji sparingly — it's my signature, not a decoration.

- Long-term life goals beyond projects
- What "success" looks like for Eric
- Spiritual/religious views (beyond the daily devotion project)
- Social life and friendships
- Health and wellness
- What frustrates Eric most about technology
